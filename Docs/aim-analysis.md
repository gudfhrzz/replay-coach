# 에임 분석 도메인 확장 조사 (2026-07-16)

경제 판단(바이 결정)에 이어 두 번째 분석 도메인 후보로 **에임/듀얼 역학**
(프리에이밍, 스프레이, 감도 힌트)의 타당성을 실데모로 검증한 기록.

## 결론 요약

- **크로스헤어 플레이스먼트·스프레이 정확도는 demoparser2만으로 계산 가능** —
  ESL 프로 데모(vitality vs spirit, nuke, 462MB)로 스파이크 검증 완료.
- 결과가 실제 평판과 일치: 크로스헤어 오프셋 최상위 = ropz(1.3°)·ZywOo(1.6°),
  스프레이 정확도는 앵커 롤(zont1x·magixx 35%)이 엔트리(apEX 7%)보다 높음.
- **감도(빠름/느림) 자체는 데모에서 읽을 수 없음** — 데모에는 시점각만 있고
  마우스 입력·인게임 감도 설정이 없다. 오버슈트/언더슈트 패턴으로 간접 추정만
  가능 (실험적, 검증 필요 — 아래 참고).
- 아키텍처 관점: 경제 판단과 동일한 패턴(조건 셀 → 프로 분포 → 유저 비교 →
  LLM 리뷰)으로 흡수 가능. "교전(engagement)"을 새 결정 지점 타입으로 모델링.

## 데모에서 뽑을 수 있는 것 (검증됨)

demoparser2 기준, 이번 스파이크에서 실제 확인한 것만:

| 데이터 | API | 비고 |
|---|---|---|
| 발사 이벤트 | `parse_event("weapon_fire")` | tick, 무기 (`weapon_` 접두사 있음) |
| 피해 이벤트 | `parse_event("player_hurt")` | 공격자, 피해량, **hitgroup(head 등)**, 무기 (접두사 없음 — 주의) |
| 킬 이벤트 | `parse_event("player_death")` | 공격자/피해자/어시스트 |
| 틱 단위 시점각 | `parse_ticks(["pitch","yaw"], ticks=[...])` | 원하는 틱만 골라 파싱 → 462MB 데모도 수 초 |
| 틱 단위 좌표 | `X, Y, Z` | 눈높이는 근사 필요 (서있음 +64, 앉음은 다름) |
| 상태 플래그 | `spotted, is_scoped, flash_duration, active_weapon_name` | spotted는 서버 판정 근사 |

## 후보 지표 (스파이크로 검증한 것 → v0 후보)

### 1. 크로스헤어 플레이스먼트 (프리에이밍) — 검증됨
킬/교전 0.25s 전 시점각과 "상대 머리 방향 각도"의 차이(도 단위).
**pitch/yaw 성분 분해가 코칭 포인트**: 세로 오차가 크면 "크로스헤어가 낮다/높다",
가로 오차가 크면 "앵글 프리에임이 안 돼 있다"로 바로 행동 가능한 조언이 된다.
프로 분포 셀 후보: (맵, 지역, 무기 클래스) — 표본 확보량에 따라 축소.

### 2. 스프레이 정확도 — 검증됨
3발+ 연속 사격(발사 간격 ≤12틱)의 명중률. hitgroup으로 헤드샷 비중도 분리 가능.
프로 분포 셀 후보: (무기, 교전 거리 구간).

### 3. 감도 힌트 (오버슈트/언더슈트) — 실험적, 미정
데모에 감도 설정이 없으므로 직접 측정 불가. 간접 추정 아이디어:
- 플릭 직후 **반대 방향 보정**이 반복되면 오버슈트 → 감도가 높을 가능성.
- 같은 방향 **추가 보정(끌기)**이 반복되면 언더슈트 → 감도가 낮을 가능성.
- 스파이크에서 "킬 직전 0.25s 시점 이동량"까지는 계산했으나(중앙값 1.6~3.9°),
  오버/언더슈트 판정 로직과 임계값은 미구현. **유저 데모 여러 개로 안정성 검증
  전에는 코칭 문구에 쓰지 않는다** (오진단 리스크).

### 4. 기타 후보 (미착수)
- 첫 발 명중률(교전 시작 무기별), 카운터 스트레이핑(발사 순간 속도),
  섬광 맞은 상태 교전 회피 여부(`flash_duration`).

## 한계·주의점

- **가시성 판정**: "적이 보이기 시작한 시점"의 엄밀한 계산은 맵 지오메트리
  레이캐스트가 필요. v0은 `spotted` 플래그나 킬 기준 고정 오프셋(0.25s 전)
  근사로 시작 — Leetify류도 같은 계열의 근사를 쓴다.
- **눈높이 근사**: 앉기/이동 자세 반영 안 하면 pitch 오차 소량 발생.
- **표본 크기**: 매치 1개당 플레이어별 킬 10~25개 수준 → 유저 리뷰는 단일 매치로
  가능하지만, 프로 분포 DB는 매치 수백 개 필요 (경제 DB와 동일한 상황).
- **선행 사례와 차별화**: Leetify·NextFrag가 크로스헤어 플레이스먼트/TTD/스프레이
  정확도를 이미 제공. 우리 차별점은 기존 방향 그대로 — **같은 상황의 프로 분포와
  비교 + LLM이 트레이드오프를 설명 + (장기) 티어 맥락화**. 지표 자체보다
  "그래서 뭘 고칠지"를 말해주는 레이어가 제품.

## 아키텍처 적합성

파이프라인 원칙(파서만 게임별, 이후 게임 무관) 유지 가능:

```
.dem → parser/cs2/duel.py (신규) → DecisionPoint(kind="engagement")
     → 프로 분포 DB (조건 셀: 맵/무기/거리) → 비교 엔진 → LLM 리뷰
```

- 경제 판단이 "라운드 단위 팀 결정"이라면 교전은 "이벤트 단위 개인 결정+실행".
  DecisionPoint 스키마의 context/action/outcome 구조에 들어가는지 확인 필요 —
  안 맞으면 스키마 확장(예: `metrics` 필드)이 논점. **미정.**
- 파싱 비용: 이벤트는 즉시, 틱은 필요한 틱만 골라 파싱하면 462MB 데모도 수 초.
  웹 v0의 동기 처리로도 감당 가능한 수준.

## 미정 사항

- 에임 도메인을 MVP에 포함할지, 경제 v1 이후 2차 도메인으로 미룰지.
- DecisionPoint 스키마 확장 방식 (`kind="engagement"` vs 별도 metrics 트랙).
- 감도 힌트의 코칭 노출 여부 (검증 전 노출 금지 원칙만 확정).
- 프로 분포 셀 정의 (맵×지역×무기 → 표본 부족 시 어디까지 뭉갤지).
